# Validation Report

## Scope

This report documents direct QEMU validation of the current `codex64` implementation and compares the observed behavior with the intended system goals.

Date of validation:

- March 13, 2026

Environment:

- `qemu-system-x86_64` 10.2.0
- BIOS boot path
- Raw disk image generated by `make all`

## Commands Used

Build:

```sh
make all
```

Serial QEMU run:

```sh
make run
```

Windowed QEMU run target added:

```sh
make run-window
```

## Test Sequence 1: Boot And Shell

Observed boot output:

```text
codex64 kernel booted
memory=133692416 bytes regions=7
codex64>
```

What this proves:

- BIOS stage 1 and stage 2 both execute correctly
- The machine enters 64-bit long mode successfully
- The kernel entry point is reached
- Serial and VGA console initialization succeeds
- The BIOS E820 map is handed into the kernel and parsed
- The shell starts and accepts input

## Test Sequence 2: Operator Commands

Commands executed:

```text
help
uname
uptime
fsinfo
meminfo
```

Observed results included:

- Command list printed correctly
- Kernel identity printed as `codex64 0.3 x86_64 monolithic research kernel`
- PIT uptime tick counter was nonzero
- Filesystem reported online with version 2
- Memory information reported total, used, free, page faults, region count, and allocation window

What this proves:

- The shell dispatcher works
- PIT timekeeping works
- The ATA-backed trace filesystem mounted successfully
- The physical memory allocator initialization is stable enough to report real bounds

## Test Sequence 3: Trace Recording

Commands executed:

```text
trace start
attack_sim
sysload
trace stop
trace list
trace stats
proc
irqstat
traceview 1
replay session 1
```

Observed highlights:

- `attack_sim` spawned PID 2
- `sysload` spawned PIDs 3, 4, and 5
- `trace list` reported session 1 with 16 events and 640 bytes
- `trace stats` reported zero dropped events
- `proc` showed live kernel tasks with per-task state and counters
- `irqstat` showed PIT interrupts arriving on vector 32
- `traceview 1` showed a mixed event stream including shell, IRQ, scheduler, and workload events
- `replay session 1` reported:

```text
match=100%
```

What this proves:

- The scheduler can spawn and rotate kernel tasks
- Scheduler events are being recorded
- Workload tasks produce trace traffic
- Timer interrupts are active during runtime
- Trace sessions are persisted to disk when recording stops
- Sequence hashing and replay-side validation are functioning

## Test Sequence 4: Persistence Across Reboot

After terminating QEMU and booting again from the same raw image, the following commands were executed:

```text
ls
fsinfo
traceview 1
replay session 1
```

Observed results:

- `ls` listed `session-1.bin`
- `fsinfo` reported one trace file and an advanced `next_free_lba`
- `traceview 1` loaded the saved session and displayed the same 16 events
- `replay session 1` again reported a `100%` hash match

What this proves:

- Trace sessions are genuinely disk-persistent, not in-memory only
- The on-disk inode and extent metadata are consistent across reboot
- Replay validation uses persisted trace content successfully

## Test Sequence 5: Deterministic Replay Demo Path

Commands executed for single-task replay:

```text
demo record attack
trace list
demo replay 1
```

Observed result:

- session 1 stored with `profile=1`
- `demo replay 1` reran the `attack` workload
- replay completed with:

```text
replay executed session=1 matched_events=50/50 profile=1
```

Commands executed for multi-task replay:

```text
demo record sysload
trace list
demo replay 1
```

Observed result:

- session 1 stored with `profile=2`
- three worker tasks were respawned deterministically
- replay completed with:

```text
replay executed session=1 matched_events=228/228 profile=2
```

What this proves:

- replay mode is no longer only passive trace inspection
- the kernel can suppress live scheduling freedom and force recorded scheduler choices for built-in deterministic workload profiles
- virtual replay time and event-by-event comparison are active
- divergence reporting infrastructure exists, even though the successful runs did not trigger it

## Test Sequence 6: Process Lifecycle And Introspection

Commands executed:

```text
state
explain kernel
explain proc
lifecycle_test
proc
demo record lifecycle
demo replay 1
proc
```

Observed results:

- `state` printed live kernel status, current PID, replay flags, and workload activity
- `explain kernel` and `explain proc` described the subsystem behavior from the shell
- `lifecycle_test` spawned three child processes
- `proc` showed lifecycle states for those children
- after execution, children appeared as `zombie`
- `demo record lifecycle` and `demo replay 1` completed with:

```text
replay executed session=1 matched_events=32/32 profile=3
```

What this proves:

- the kernel is now multi-process capable in the current single-core design
- process lifecycle states are explicit and inspectable
- built-in subsystem introspection is available from the shell
- the process-lifecycle workload is replayable under the current deterministic demo mode

## Test Sequence 7: Input Fixes And SMP Groundwork

Input checks:

- serial backspace/delete was tested by editing a command line in place
- underscore input was verified through the shell path
- keyboard handling was updated to track shift state and produce shifted symbols such as `_`

SMP groundwork checks:

Default QEMU boot:

```text
smpinfo
```

Observed result:

```text
smp enabled=1 discovered=1 online=1 current_cpu=0 ...
```

Direct QEMU validation with two vCPUs:

```sh
qemu-system-x86_64 -smp 2 ...
```

Observed result from `smpinfo`:

```text
smp enabled=1 discovered=2 online=1 current_cpu=0 ...
cpu slot=0 apic_id=0 online=1 bsp=1

```

What this proves:

- the shell now handles backspace/delete correctly
- shifted keyboard symbols are implemented in the PS/2 keyboard path
- CPUID/APIC/SMP discovery groundwork is working
- the kernel can see multi-vCPU topology and represent offline secondary CPUs, even though only the BSP is currently active

## Requirement Mapping

### Strongly demonstrated

- 64-bit boot pipeline
- Monolithic kernel bring-up
- Shell interface
- Interrupt controller and PIT support
- PS/2 keyboard input path in the kernel
- VGA and serial output
- Basic physical memory management
- Kernel task scheduling
- Persistent trace storage
- Trace listing, viewing, and replay-summary commands

### Partially demonstrated

- Replay subsystem
  Current implementation validates stored event sequences and hashes, but does not yet inject events to force an exact control-flow reenactment.
- Exploit-workload reproduction
  `attack_sim` is a synthetic exploit-like workload rather than a full privilege-escalation test case.
- Filesystem
  Persistent and functional for trace sessions, but not yet a general-purpose Unix-like filesystem.

### Not yet fully implemented

- Full syscall layer
- Real user processes with separate address spaces
- Full deterministic replay with interrupt suppression/injection
- Blocking/nonblocking filesystem APIs integrated with a preemptive scheduler
- Rich driver set beyond the current minimum hardware path

## Why The Current Result Still Matters

Even with the remaining gaps, the present build already demonstrates the core thesis at a prototype level:

- The kernel can capture execution-relevant events
- Those events can be stored persistently
- The saved session can survive reboot
- The session can be reloaded and validated for exact sequence integrity

That is enough to show that the OS is functioning as a replay-oriented experimental platform, not just as a boot demo.

## Conclusion

The system works as a real bootable 64-bit hobby kernel with:

- boot
- shell
- interrupts
- scheduler
- memory accounting
- ATA-backed persistent trace storage
- trace inspection
- replay-sequence validation

The tests do support the claim that the current implementation matches the present prototype goals and is materially aligned with the larger research direction. The main thing not yet proven is full deterministic execution replay in the strictest sense. That remains the next major engineering step, not a verified property of the current build.
